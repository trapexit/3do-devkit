## Introduction

This document defines **ARM Object Format (AOF)**, used by language processors for ARM-based systems.

The AOF linker accepts input files in this format and produces output either in AOF or in RISC OS Application Image Format (AIF; Technical Memorandum PLG-AIF).

In this document:

- *object file* means a file in AOF format;
- *linker* means the AOF linker.

## Assumed Terminology

### Byte

8 bits, considered unsigned unless otherwise stated, usually used for flags or characters.

### Half Word

16 bits (2 bytes), usually unsigned, little-endian. The address of a half word must be divisible by 2.

### Word

32 bits (4 bytes), usually non-negative, little-endian. The address of a word must be divisible by 4.

### String

A sequence of bytes terminated by a NUL (`0x00`) byte. The NUL is part of the string but not counted in its length. Strings may be aligned on any byte boundary.

Emphasis from the original text: AOF words are 32-bit, 4-byte aligned, and use DEC/Intel little-endian byte order, not IBM/Motorola big-endian byte order.

## Undefined Fields

Fields not explicitly defined are reserved to Acorn and are required to be zero. Acorn may define future meanings, usually preserving the meaning of zero values.

## Overall Structure of an AOF File

An AOF object file is layered over **Chunk File Format** to provide efficient access and update of multiple logical data pieces in one file.

Defined AOF chunks:

- Header
- Areas
- Identification
- Symbol Table
- String Table

Minimum data unit in both formats is one word (4 bytes), stored little-endian.

## Chunk File Format

The file begins with a chunk header describing the number, identity, size, and location of each chunk.

```text
         ----------------------------
         |       ChunkFileId        |
         ----------------------------
         |       maxChunks          |
         ----------------------------
         |       numChunks          |  3 words
         ============================
         |        entry1            |  4 words per entry
         |                          |
         ----------------------------
         |        entry2            |
         |                          |
         ----------------------------
                   ...
         ----------------------------
         |     entry maxChunks      |  End of Header
         |                          |  (3 + 4*maxChunks words)
         ============================
         |        chunk 1           |  Start of Data Chunks
         |                          |
         ----------------------------
                   ...
         ----------------------------
         |      chunk numChunks     |
         |                          |
         ----------------------------
```

- `ChunkFileId` = `0xC3CBC6C5`.
- `maxChunks` is fixed at file creation.
- `numChunks` is current chunk count (`0..maxChunks`), redundant but stored.

Each chunk table entry has four words:

- `chunkId` (2 words): identifies chunk contents.
- `fileOffset` (1 word): byte offset of chunk data; divisible by 4; `0` means unused entry.
- `size` (1 word): exact chunk byte size (need not be multiple of 4).

`chunkId` naming convention:

- first 4 chars: globally allocated namespace (for AOF: `OBJ_`);
- second 4 chars: per-namespace component identifier.

Characters are stored in forward order as text bytes.

## Object File Format

AOF defines five chunk names:

| Chunk | Chunk Name |
|---|---|
| Header | `OBJ_HEAD` |
| Areas | `OBJ_AREA` |
| Identification | `OBJ_IDFN` |
| Symbol Table | `OBJ_SYMT` |
| String Table | `OBJ_STRT` |

Only `OBJ_HEAD` and `OBJ_AREA` are mandatory.

Chunks may appear in any order. Additional language/tool chunks may also be present; reserving space for about 8 chunk entries is described as conventional.

## Format of the AOF Header Chunk (`OBJ_HEAD`)

The AOF header has:

1. a fixed part (6 words), and
2. a variable-length sequence of area declarations (5 words each).

```text
         ----------------------------
         |    Object File Type      |
         ----------------------------
         |       Version Id         |
         ----------------------------
         |    Number of Areas       |
         ----------------------------
         |    Number of Symbols     |
         ----------------------------
         |    Entry Address Area    |
         ----------------------------
         |    Entry Address Offset  |  6 words in fixed part
         ============================
         |    1st Area Header       |  5 words per area
         |                          |
         ----------------------------
         |    2nd Area Header       |
         |                          |
         ----------------------------
                  ...
         ----------------------------
         |    nth Area Header       |
         |                          |
         ----------------------------
```

### Fixed fields

- **Object File Type**: `0xC5E2D080` marks relocatable object format.
- **Version Id**: AOF 1.xx uses decimal `150`; AOF 2.xx uses decimal `200`.
- **Number of Areas**: count of area declarations and areas in `OBJ_AREA`.
- **Number of Symbols**: symbol count if `OBJ_SYMT` is present.
- **Entry Address Area/Offset**: optional program entry as `<area-index, offset>`, where area index is `1..Number of Areas`. An area-index of `0` means no entry address is defined.

## Format of Area Headers

Each area header is 5 words:

```text
Area Name                       (offset into string table)
zeroes | AT | AL
Area Size
Number of Relocations
Unused (must be 0)
```

### Area Name

Offset into `OBJ_STRT`. Area names must be unique within an object file.

### AL (Area Alignment)

Must be `2`. Other values are reserved to Acorn.

### AT (Area Attributes)

Attributes are encoded in an 8-bit field. The linker orders areas by:

1. attributes,
2. case-significant lexical area name,
3. position of the containing object module in link order.

For attribute grouping, original behavior is: Read-Only before Read-Write before Debug; and within RO/RW: Code before Data before Zero-Initialised Data.

#### AT bit meanings

- **Bit 0**: must be `0`.
- **Bit 1**: `1` = code, `0` = data.
- **Bit 2**: common block definition.
- **Bit 3**: common block reference (implies bit 4; no initialising data).
- **Bit 4**: area has no initialising data in `OBJ_AREA` (zero-initialised).
- **Bit 5**: read-only; incompatible with bit 4.
- **Bit 6**: must be `0`.
- **Bit 7**: symbolic debugging tables (if set, bit 1 is ignored).

Common block rules:

- common areas with same name are overlaid;
- a definition (bit 2) gives block size;
- references must not exceed definition size;
- in a link step there may be at most one definition (bit 2 set) for a name;
- if none defined, final size is largest reference.

### Area Size

Area size in bytes, multiple of 4. Unless bit 4 (not initialised) is set, this many bytes must appear in `OBJ_AREA`.

### Number of Relocations

Count of relocation records associated with this area.

## Format of the Areas Chunk (`OBJ_AREA`)

`OBJ_AREA` stores each area's bytes followed by its relocation table. Both are word-aligned.

```text
             -----------------------
             |       Area 1        |
             -----------------------
             |  Area 1 Relocation  |
             -----------------------
                      ...
             -----------------------
             |       Area n        |
             -----------------------
             |  Area n Relocation  |
             -----------------------
```

An area is either fully initialised from file bytes or not initialised at all (and treated as zero-initialised in loaded images), according to area attribute bit 4.

## Relocation Directives

Without relocation, stored byte/halfword/word values are final image values.

- Bytes and halfwords may be relocated only by suitably small constant values.
- Bytes and halfwords may not be relocated by an area's base address.
- A field may have more than one relocation directive.

Two relocation directive formats are defined:

- **Type-1**
- **Type-2** (AOF version 150 and later)

Relocations are also classified by behavior:

- **Additive**
- **PCRelative**

For additive and type-2 PC-relative relocations, there are:

- **Internal** variant
- **Symbol** variant

### Type-1 Relocation Directive

```text
             -----------------------
             |       Offset        |
             -----------------------
             | 0 |A|R|FT|   SID    |
             -----------------------
```

- `Offset`: byte offset of field in preceding area.
- `SID` (16-bit): symbol-table index when symbol relocation is used.
- `FT` (bits 16-17): field type:
  - `00` byte
  - `01` halfword
  - `10` word
  - `11` illegal
- `R` (bit 18): `0` additive, `1` PC-relative.
- `A` (bit 19): interpreted only when `R=0`:
  - `0` internal (add base address of current area)
  - `1` symbol (add symbol value)
- bits 20-31: reserved, write as zero.

Type-1 PC-relative relocations always reference a symbol. Linker accounts for ARM PC being 8 bytes beyond the branch instruction.

### Type-2 Relocation Directive (AOF 150+)

```text
             ----------------------------
             |         Offset           |
             ----------------------------
             | 1000 |A|R|FT| 24-bit SID |
             ----------------------------
```

- Top bit of second word is `1`.
- Bits 28..30 must be zero.
- `SID` is 24-bit.
- `Offset` and `FT` meanings match type-1.

If `A=0` (internal), `SID` is area index (0-origin, in `OBJ_AREA`) relative to which relocation is performed.

The document explains type-2 PC-relative relocation as compatible with translators that also emit Unix `a.out`:

- instruction offset is initially based on area-base relative PC assumptions;
- symbol/internal relocation then adjusts by symbol or referenced area offset relative to the containing area's base.

The linker may emit type-2 internal PC-relative relocations when partially linking object modules.

## Format of the Symbol Table Chunk (`OBJ_SYMT`)

`Number of Symbols` in `OBJ_HEAD` gives the symbol count. Each symbol entry is 4 words:

```text
Name
AT (7-bit attributes field in low bits)
Value
Area Name
```

### Name

Offset into `OBJ_STRT` for the symbol string.

### Symbol attributes (AT)

#### Bits 1..0

- `01`: local definition (defined in this object file, file-local scope).
- `10`: external reference.
- `11`: global definition.
- `00`: reserved.

If an external reference (`10`) is unresolved, linker also attempts match by common area names.

#### Other bits

- **Bit 2**: absolute value (meaningful for definitions).
- **Bit 3**: case-insensitive reference (meaningful for external refs).
- **Bit 4**: weak external reference (may remain unresolved; relocated fields may remain unrelocated).
- **Bit 5**: strong global definition (used to override non-strong definitions from other files in external resolution).
- **Bit 6**: common symbol reference; `Value` gives common area length.

Common symbol behavior:

- common symbols with same name share one allocated base;
- allocated size is max requested size;
- if matching common area exists, symbol and area are merged;
- unmatched common symbols are placed in anonymous linker pseudo-area.

### Value

Meaningful when symbol is a definition (bit 0 set) or common symbol (bit 6 set):

- if absolute (bit 2 set): literal symbol value;
- otherwise: offset from base of named logical area.

### Area Name

Meaningful only for non-absolute definitions (bit 0 set, bit 2 unset). Contains offset in `OBJ_STRT` of the logical area name used as the base for `Value`.

## The String Table Chunk (`OBJ_STRT`)

Contains names used by areas and symbols.

- Each name: ISO8859 non-control characters + trailing NUL.
- Names are referenced by offset from table start.
- First 4 bytes are table length (including those 4 bytes).
- Therefore, valid name offsets are `>= 4`.
- The stored length must match the chunk header length for `OBJ_STRT`.

## The Identification Chunk (`OBJ_IDFN`)

Contains a NUL-terminated text string describing the name/version of the generating language translator. Non-whitespace control characters are excluded.

## Linker-Defined Symbols (Informative)

Not part of AOF encoding itself, but defined by the AOF linker when building images. These symbols may be referenced from AOF object files and must not be redefined.

### Pre-defined image symbols

All start with `Image$$` and are reserved to Acorn. They are Base/Limit pairs where Base is first byte of region and Limit is first byte beyond region.

- `Image$$RO$$Base`, `Image$$RO$$Limit`
- `Image$$RW$$Base`, `Image$$RW$$Limit`
- `Image$$ZI$$Base`, `Image$$ZI$$Limit`

Notes from the original:

- If a section is absent, Base and Limit are equal but unpredictable.
- `Image$$RO$$Base` includes any linker-prepended image header.
- `Image$$RW$$Limit` includes zero-initialised data created at runtime at the RW end.
- `Image$$ZI$$*` includes zero-initialised areas and unmatched common symbols.

### Common area symbols

For each common area, linker defines a global symbol with the same name unless it clashes with an existing global symbol definition.

## Obsolescent and Obsolete Features

The 2.00/C document records previously supported features no longer supported.

### Obsolete object file types

Historically defined image types:

- `0xC5E2D081` (AOF image type 1, obsolescent)
- `0xC5E2D083` (AOF image type 2, obsolete)
- `0xC5E2D087` (AOF image type 3, obsolete)

### Area alignment (`AL`)

Older revisions allowed powers of 2 from 2 to 16. In current usage all values other than `2` are reserved.

### Withdrawn area attributes (`AT`)

Withdrawn from earlier revisions:

- Absolute attribute (old bit 0)
- Position Independent attribute (old bit 6)

Rationale given: limited utility in RISC OS linker and relationship to obsolete image types.

### Fragmented areas

Fragmented areas were introduced in AOF 0.04 but are declared withdrawn in this issue.
