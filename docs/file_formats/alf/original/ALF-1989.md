## Introduction

A library file contains a number of separate but related pieces of data.
In order to simplify access to these data, and to provide for a degree of
extensibility, the library file format is itself layered on another format
called **Chunk File Format**, which provides a simple and efficient means of
accessing and updating distinct chunks of data within a single file.

The library file format defines four chunks: `Directory`, `Time-stamp`,
`Version`, and `Data`. There may be many `Data` chunks in a library.

The minimum size of a piece of data in both formats is four bytes (one word).
Each word is stored in a file in little-endian format; that is, the least
significant byte of the word is stored first.

## Chunk File Format

A chunk is accessed via a header at the start of the file. The header contains
the number, size, location, and identity of each chunk in the file. The size
of the header may vary between different chunk files but is fixed for each
file.

Not all entries in a header need be used, so limited expansion of the number
of chunks is permitted without a wholesale copy. A chunk file can be copied
without knowledge of the contents of the individual chunks.

Graphically, the layout of a chunk file is as follows:

```text
             ----------------------------
             |       ChunkFileId        |
             ----------------------------
             |       maxChunks          |
             ----------------------------
             |       numChunks          |  3 words
             ============================
             |        entry1            |  4 words per entry
             |                          |
             ----------------------------
             |        entry2            |
             |                          |
             ----------------------------
                       ...
             ----------------------------
             |     entry "maxChunks"    |  End of Header
             |                          |  (3 + 4*maxChunks words)
             ============================
             |        chunk  1          |  Start of Data Chunks
             |                          |
             ----------------------------
                       ...
             ----------------------------
             |    chunk  "numChunks"    |
             |                          |
             ----------------------------
```

`ChunkFileId` marks the file as a chunk file. Its value is `0xC3CBC6C5`.

- `maxChunks`: number of entries in the header, fixed when the file is created.
- `numChunks`: number of chunks currently used in the file, varying from `0`
  to `maxChunks`. (`numChunks` is redundant because it can be found by scanning
  the entries.)

Each entry in the header comprises four words in the following order:

- `chunkId`: a two-word field identifying what data the chunk contains.
- `fileOffset`: a one-word field defining the byte offset within the file of
  the chunk (must be divisible by four). An entry of zero indicates the
  corresponding chunk is unused.
- `size`: a one-word field defining the exact byte size of the chunk (need not
  be a multiple of four).

The `chunkId` field provides a conventional way of identifying what type of
data a chunk contains. It is split into two parts:

- First 4 characters (first word): a universally unique name allocated by a
  central authority (Acorn).
- Remaining 4 characters (second word): used to identify component chunks
  within this universal domain.

In each part, the first character of the name is stored first in the file.

For library files, the first part of each chunk's name is `LIB_`.

## Library File Format Types

Two library file formats are described: **new-style** and **old-style**.

- The linker and library management tools can read both formats.
- No tool generates old-style libraries.
- At the time of writing, only the Fortran-77 compiler generates old-style
  libraries (instead of AOF object files).
- The linker handles old-style libraries specially, including every member in
  the output image unless explicitly instructed otherwise.

Old-style libraries are obsolescent and should no longer be generated.

## Library File Format

Each piece of a library file is stored in a separate identifiable chunk:

| Chunk | Chunk Name |
|---|---|
| Directory | `LIB_DIRY` |
| Time-stamp | `LIB_TIME` |
| Version | `LIB_VSRN` (new-style only) |
| Data | `LIB_DATA` |
| Symbol table | `OFL_SYMT` (object code libraries only) |
| Time-stamp | `OFL_TIME` (object code libraries only) |

There may be many `LIB_DATA` chunks in a library, one for each library member.

### LIB_DIRY

The `LIB_DIRY` chunk contains a directory of all modules in the library, each
of which is stored in a `LIB_DATA` chunk.

- Directory size is fixed when the library is created.
- The directory consists of a sequence of variable-length entries.
- Each entry is an integral number of words long.
- Number of directory entries is determined by size of `LIB_DIRY`.

Pictorial layout:

```text
                    +-> +--------------------+
                    |   |    ChunkIndex      |
                    |   +--------------------+
                    +------- EntryLength     |
                    |   +--------------------+
                 an |   |    DataLength ---------+
           integral |   +--------------------+ <-+
             number |   |                    |   | in an old-style library,
                 of |   .    Data            .   | may be an odd number of
              words |   .                    .   | bytes
                    |   .              /-----+ <-+
                    |   +-------------/      .
                    |   |    Padding         |
                    +-> +--------------------+
```

#### ChunkIndex

The `ChunkIndex` is a 0-origin index within the chunk file header of the
corresponding `LIB_DATA` chunk.

- The `LIB_DATA` chunk entry gives the offset and size of the library module.
- A `ChunkIndex` of `0` means the directory entry is not in use.

#### EntryLength

Number of bytes in this `LIB_DIRY` entry; always a multiple of 4.

#### DataLength

Number of bytes used in the Data section of this `LIB_DIRY` entry.

- Need not be a multiple of 4.
- In new-style libraries, it always is a multiple of 4.

#### Data

The data section consists of a NUL-terminated string followed by any other
information relevant to the library module.

- Strings should contain only ISO-8859 non-control characters (codes
  `[0-31]`, `127`, and `128+[0-31]` excluded).
- The string is the name used by library management tools to identify the
  module (typically source filename).

In new-style libraries, an 8-byte word-aligned time-stamp follows the member
name. Format is described in `LIB_TIME`. Its value is an encoded version of
the file timestamp (last modified time) of the source file.

Writers should ensure `LIB_DIRY` entries contain valid time-stamps. Readers
should not rely on data beyond the name string unless `DataLength` allows for
it. Even then, time-stamp contents should be treated cautiously.

Applications writing `LIB_DIRY` or `OFL_SYMT` entries should pad with NUL
(`0`) bytes; readers should make no assumptions about padding bytes beyond the
first string-terminating NUL.

### LIB_TIME

The `LIB_TIME` chunk contains a 64-bit time-stamp recording when the library
was last modified, in this format:

```text
  High-address byte      low-address byte
         |                    |
        +----------------+-----+
        |    TimeStamp   |     |
        +----------------+-----+
                .           .
               /|\         /|\
                |           +--- 2 byte micro second count, usually 0
                |
                +--------------- 6 bytes of centi-seconds since
                                                   1/1/1900 00:00 GMT
```

### LIB_VSRN

In new-style libraries, this chunk contains a 4-byte version number.

- Current version number: `1`
- Old-style libraries do not contain this chunk.

### LIB_DATA

A `LIB_DATA` chunk contains one library member indexed by `LIB_DIRY`.

The library management tools do not interpret member contents. A member could
itself be a chunk file, or even another library.

## Object Code Libraries

An object code library is a library file whose members are files in
Acorn Object Format (AOF; see PLG-AOF).

Additional information is stored in two extra chunks: `OFL_SYMT` and
`OFL_TIME`.

`OFL_SYMT` contains an entry for each external symbol defined by library
members, together with the index of the chunk containing the member that
defines that symbol.

The `OFL_SYMT` chunk has exactly the same format as `LIB_DIRY` except that:

- Data section of each entry contains only a symbol name string.
- Followed by between 1 and 4 bytes of NUL padding.
- `OFL_SYMT` entries do not contain time-stamps.

The `OFL_TIME` chunk records when the `OFL_SYMT` chunk was last modified and
has the same format as `LIB_TIME`.
