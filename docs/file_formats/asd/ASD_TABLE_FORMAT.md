# ARM Symbolic Debug Table (ASD) Format Specification

**Consolidated Technical Reference**

This document describes the ARM Symbolic Debug Table (ASD) payload format in a form intended for parser, dumper, converter, and writer implementation.

Sources used:

- `original/arm_dui_0041b_chapter_15_-_arm_symbolic_debug_table_format.pdf`
- `original/ASDTF.txt`

This document covers the ASD debug-table data itself. The enclosing object or image format is responsible for locating the ASD data block, defining target endianness, and applying relocations to address fields.

---

## 1. Purpose And Model

ASD tables contain symbolic debugging data generated by ARM compilers and consumed by the ARM Symbolic Debugger for high-level, source-oriented debugging.

For each separate compilation unit, called a **section** by the ASD specification, the compiler emits debugging data and a corresponding object-code debug area. The linker combines these debug areas into one contiguous debug data block in the final image.

ASD also supports low-level assembly/symbol debugging. Low-level tables are generated by the linker and appear as an extra ASD section, as if they came from an independent compilation unit. Low-level and high-level debugging are independent facilities; a program image can contain either or both.

Important design properties:

- ASD item-to-item references are byte offsets relative to the start of the current ASD section.
- Code and data references are normal relocatable address fields supplied by the enclosing object/image format.
- The debug data is position-independent after relocation because internal references are relative offsets.
- A debugger may copy the debug data elsewhere before the debuggee overwrites the original memory.

## 2. Primitive Types And Endianness

| Name | Size | Meaning |
| --- | ---: | --- |
| byte | 1 byte | 8 bits, usually unsigned |
| halfword | 2 bytes | 16 bits, also called a short |
| word | 4 bytes | 32 bits, often interpreted as signed for offsets or values |

All words and halfwords use the target ARM system byte order, not necessarily the host byte order. ASD can therefore be little-endian or big-endian. The enclosing object/image format determines which byte order applies.

Byte sequences, including strings and byte fields, are stored in address order and are not byte-swapped.

Implementation guidance:

- Decode every `word` and `halfword` using target endianness.
- Decode named byte fields one byte at a time in file order.
- Do not infer endianness from ASD data alone unless an outer container is unavailable and a heuristic is unavoidable.

---

## 3. Overall Layout

An ASD debug block is a concatenation of one or more ASD sections.

```text
+-------------------------------+
| Section item                   |  item code 1
+-------------------------------+
| Debug items for this section   |  source-definition order
+-------------------------------+
| Fileinfo item                  |  item code 10, final when present
+-------------------------------+
| Next section item              |  item code 1
+-------------------------------+
| ...                           |
+-------------------------------+
```

Each section begins with a **section item**. The section item's `debugsize` field gives the total byte size of that section's debug data, including the section item itself. A parser should use `debugsize` to advance to the next section in the enclosing block.

Within a high-level section:

- Items appear in source-definition order.
- Procedure/function definitions are bracketed by a procedure item and a matching endproc item.
- Nested source definitions are represented by nested item ranges.
- Top-level variables and types appear outside all procedure/endproc ranges.
- The fileinfo item, when present, is always the final item in the section.

Within a low-level linker-generated section (`lang == LANG_NONE`), the section item is followed by a raw symbol table and an AOF-style string table rather than normal ASD item headers.

---

## 4. Item Header

Every normal ASD item starts with a 32-bit **code-and-length** word:

```text
31              16 15               0
+----------------+------------------+
| length in bytes| item code        |
+----------------+------------------+
```

Decode as:

```c
uint32_t h = read_u32();
uint16_t length = h >> 16;
uint16_t code = h & 0xffff;
```

The length is the byte length of the item including this header word. Unknown item codes must be skipped using this length, which is the intended extension mechanism.

Special case: a fileinfo item can have a header length of zero if its size is too large to fit in 16 bits. This is valid only because fileinfo is the final item and the enclosing section item contains `debugsize`. In that case, the fileinfo extent is:

```c
fileinfo_length = section.debugsize - fileinfo_offset;
```

For any other item, or for an unknown item, a zero length should be treated as malformed because it prevents forward progress.

## 6. Strings

ASD string fields are length-prefixed byte strings padded to a word boundary.

```text
+--------+-------------------+------------------+
| length | length bytes data | zero padding     |
+--------+-------------------+------------------+
 1 byte        0..255 bytes      to 4-byte boundary
```

Rules:

- The length byte is in the range 0..255.
- The string bytes are not NUL-terminated.
- Padding bytes must be zero.
- Encoded string size is `align4(1 + length)`.
- An empty string is encoded as one zero length byte plus three zero padding bytes, i.e. a single zero word.

Use this helper when parsing variable-length structures:

```c
size_t asd_string_size(uint8_t n) {
    return (1 + n + 3) & ~3u;
}
```

## 7. Section-Relative Offsets And Addresses

When a field is described as an **offset in the debugging data area**, it is a byte offset from the start of the current ASD section, meaning from the first byte of that section's section item.

Common section-relative offsets include:

- `section.fileinfo`
- `procedure.endproc`
- `procedure.fileentry`
- `endproc.fileentry`
- variable-bound references in array items
- compound type references encoded in type words
- macro `body` and `argtable` references

When the same structures are mapped into debugger memory, old implementations may replace offsets with direct pointers. On disk or in image data, tooling should treat these fields as offsets unless it is explicitly consuming an already-mapped in-memory debugger structure.

When a field is described as an **address**, it is an address in the debugged program and is relocated by the linker or loader through the enclosing format. Examples include procedure code addresses, static variable addresses, file fragment code addresses, and naming-scope code addresses.

---

## 8. Source Positions

Several items contain a packed `sourcepos` word:

```text
31          22 21                         0
+-------------+---------------------------+
| char offset | line number               |
+-------------+---------------------------+
   10 bits              22 bits
```

Decode as:

```c
uint32_t char_offset = sourcepos >> 22;        // zero-based
uint32_t line_number = sourcepos & 0x003fffff;
```

The character offset is zero-based from the start of the line. The line number is stored directly in the low 22 bits.

---

## 9. Type Words

Procedure, variable, field, array, and other items identify data types with a 32-bit type word:

```text
31                         8 7                0
+---------------------------+------------------+
| signed base type code     | pointer count    |
+---------------------------+------------------+
       24 bits                    8 bits
```

Decode as:

```c
uint8_t pointer_count = type_word & 0xff;
int32_t base_code = ((int32_t)type_word) >> 8; // arithmetic shift
```

Pointer count values:

| Value | Meaning |
| ---: | --- |
| 0 | The base type itself |
| 1 | Pointer to the base type |
| 2 | Pointer to pointer to the base type |
| n | n levels of pointer indirection |

Simple base type codes are positive decimal values:

| Code | Type |
| ---: | --- |
| 0 | void |
| 10 | signed byte |
| 11 | signed halfword |
| 12 | signed word |
| 13 | signed double word |
| 20 | unsigned byte |
| 21 | unsigned halfword |
| 22 | unsigned word |
| 23 | unsigned double word |
| 30 | float |
| 31 | double |
| 32 | long double |
| 41 | single complex |
| 42 | double complex |
| 100 | function |

Compound types are represented by separate ASD items such as array, struct, union, class, bitfield, subrange, set, or enumeration items. A compound type reference uses a negative base code:

```c
if (base_code < 0) {
    uint32_t item_offset = (uint32_t)(-base_code);
}
```

`item_offset` is a byte offset from the start of the current ASD section to the item that describes the compound type.

If a source-level type has a name, a type item stores that name and a type word. The type word may directly name a simple type or may refer to a compound type item.

Limitations from the source specification:

- Pascal set types record only total object size.
- Pascal file variables are not supported by the debugger.
- Fortran character types are represented by compiler-specific debug items not defined by the base ASD specification.

## 10. Section Item, Code 1

A section item is the first item in every ASD section.

Layout for a source-level section:

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `header` | Code-and-length word, code 1 |
| 4 | 1 | `lang` | Source language code |
| 5 | 1 | `flags` | Detail flags |
| 6 | 1 | `unused` | Reserved/unused |
| 7 | 1 | `asdversion` | ASD format version |
| 8 | 4 | `codestart` | Address of first instruction in this section |
| 12 | 4 | `datastart` | Address of static data for this section |
| 16 | 4 | `codesize` | Byte size of executable code in this section |
| 20 | 4 | `datasize` | Byte size of static data in this section |
| 24 | 4 | `fileinfo` | Section-relative offset of fileinfo item, or 0 if absent |
| 28 | 4 | `debugsize` | Total byte size of this ASD section, including this item |
| 32 | variable | `name` | ASD string naming the program/module |

Layout for a low-level linker-generated section:

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `header` | Code-and-length word, code 1 |
| 4 | 1 | `lang` | `LANG_NONE` |
| 5 | 1 | `flags` | Detail flags |
| 6 | 1 | `unused` | Reserved/unused |
| 7 | 1 | `asdversion` | ASD format version |
| 8 | 4 | `codestart` | `Image$$RO$$Base` |
| 12 | 4 | `datastart` | `Image$$RW$$Base` |
| 16 | 4 | `codesize` | `Image$$RO$$Limit - Image$$RO$$Base` |
| 20 | 4 | `datasize` | `Image$$RW$$Limit - Image$$RW$$Base` |
| 24 | 4 | `fileinfo` | 0 |
| 28 | 4 | `debugsize` | Total low-level debug data size, including this item |
| 32 | 4 | `nsyms` | Number of following raw symbol records |

Language codes:

| Code | Name | Meaning |
| ---: | --- | --- |
| 0 | `LANG_NONE` | Low-level debugging data only |
| 1 | `LANG_C` | C source-level debugging data |
| 2 | `LANG_PASCAL` | Pascal source-level debugging data |
| 3 | `LANG_FORTRAN` | Fortran-77 source-level debugging data |
| 4 | `LANG_ASM` | ARM assembler line-number data |

All other language codes are reserved by ARM.

Detail flags:

| Mask | Meaning |
| ---: | --- |
| 1 | Contains line-number information |
| 2 | Contains top-level variable information |
| 3 | Contains both line-number and top-level variable information |

`asdversion` should be set to 2 according to both source documents.

For C sections, `name` is derived from the root file name and is notionally a module name. For Pascal and Fortran, `name` contains the program name. The name is similar to a source-language variable name.

### 10.1 Low-Level Symbol Records

For `LANG_NONE`, the section item is followed by `nsyms` raw symbol records. These records are not normal ASD items and do not have code-and-length headers.

Each symbol record is 8 bytes:

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `sym` | Flags plus byte offset in following string table |
| 4 | 4 | `value` | Absolute symbol value |

Decode `sym` as:

```c
uint32_t name_offset = sym & 0x00ffffff;
uint32_t sym_flags = sym & 0xff000000;
```

Flag values from the PDF source:

| Value | Name | Meaning |
| ---: | --- | --- |
| `0x00000000` | `ASD_ABSSYM` | Symbol is absolute |
| `0x01000000` | `ASD_GLOBSYM` | Symbol is global |
| `0x02000000` | `ASD_TEXTSYM` | Symbol names code |
| `0x04000000` | `ASD_DATASYM` | Symbol names data |
| `0x06000000` | `ASD_ZINITSYM` | Symbol names zero-initialized data |
| `0x10000000` | `ASD_16BITSYM` | Bit set if the symbol is a Thumb symbol |

The linker reduces all symbol values to absolute values. The flags record the symbol's history or origin in the image. `ASD_16BITSYM` is a bit flag and may be combined with an origin value.

Immediately after the symbol table is an AOF-style string table:

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `length` | Total string table byte length, including this word |
| 4 | variable | strings | NUL-terminated strings |

The length is a multiple of four. No valid string offset is less than 4. The end of the string table is padded with NUL bytes to the next word boundary.

## 11. Procedure/Function Definition Item, Code 2

A procedure item appears once for each source procedure or function definition. Any definitions nested within the procedure appear between this item and its matching endproc item.

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `header` | Code-and-length word, code 2 |
| 4 | 4 | `type` | Return type if this is a function, otherwise 0 |
| 8 | 4 | `args` | Number of arguments |
| 12 | 4 | `sourcepos` | Source position of procedure start |
| 16 | 4 | `startaddr` | Address of first instruction of the procedure prologue |
| 20 | 4 | `entry` | Address of first instruction after the prologue |
| 24 | 4 | `endproc` | Section-relative offset of the matching endproc item, or 0 for a label |
| 28 | 4 | `fileentry` | Section-relative offset of the source file entry containing the procedure start |
| 32 | variable | `name` | ASD string |

`startaddr` is the instruction where control arrives when the procedure is called. `entry` is the first instruction where a high-level breakpoint can sensibly be set.

Labels are represented as special procedure items with `endproc == 0`. Pascal and Fortran numerical labels are converted to strings prefixed by `$n`.

For Fortran-77, multiple entry points to the same procedure produce separate procedure items that all refer to the same endproc item.

---

## 12. Endproc Item, Code 3

An endproc item marks the end of the debug items belonging to a procedure and records procedure-return information.

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `header` | Code-and-length word, code 3 |
| 4 | 4 | `sourcepos` | Source position of procedure end |
| 8 | 4 | `endpoint` | Address of the code byte after the compiled procedure |
| 12 | 4 | `fileentry` | Section-relative offset of the source file entry containing the procedure end |
| 16 | 4 | `nreturns` | Number of procedure return points, possibly 0 |
| 20 | `4*nreturns` | `retaddrs` | Array of addresses of procedure return code |

If the procedure body is an infinite loop, `nreturns` is 0. Otherwise, each return address should identify a suitable exit-breakpoint location where the current stack frame still belongs to the procedure.

---

## 13. Variable Item, Code 4

A variable item describes a source variable or a formal procedure/function argument. The first variable items inside a procedure always describe the procedure's arguments.

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `header` | Code-and-length word, code 4 |
| 4 | 4 | `type` | Type word |
| 8 | 4 | `sourcepos` | Source position of the variable definition |
| 12 | 4 | `storageclass` | Storage class code |
| 16 | 4 | `location` | Location, interpreted by storage class |
| 20 | variable | `name` | ASD string |

Storage class codes:

| Code | Meaning |
| ---: | --- |
| 1 | External variable, or Fortran common |
| 2 | Static variable private to one section |
| 3 | Automatic variable |
| 4 | Register variable |
| 5 | Pascal `var` argument |
| 6 | Fortran argument |
| 7 | Fortran character argument |

`location` interpretation:

| Storage class | Location meaning |
| --- | --- |
| External or static | Absolute address, relocated by the linker |
| Automatic | Stack offset from frame pointer |
| Pascal `var` argument | Stack offset from frame pointer |
| Fortran argument | Offset into argument list |
| Register variable | Register number |

For register variables, ARM integer registers use their normal register numbers. The eight floating-point registers are numbered 16 through 23.

The `sourcepos` field lets a debugger distinguish otherwise identical names in disjoint source scopes, such as C nested blocks.

The text source notes that no account is taken of variables that should be addressed by positive offsets from the stack pointer rather than negative offsets from the frame pointer. Later FP map fragment items provide virtual frame pointer information for functions without a physical frame pointer.

A type item describes a named source-language type, such as a C `typedef`.

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `header` | Code-and-length word, code 5 |
| 4 | 4 | `type` | Type word |
| 8 | variable | `name` | ASD string |

## 15. Struct, Class, And Union Items, Codes 6, 19, 20

Struct, class, and union items have the same binary format.

| Code | Item |
| ---: | --- |
| 6 | Struct |
| 19 | Class |
| 20 | Union |

Aggregate item layout:

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `header` | Code-and-length word |
| 4 | 4 | `fields` | Number of fields |
| 8 | 4 | `size` | Total byte size of the aggregate |
| 12 | variable | `fieldtable` | Sequence of `fields` field descriptors |

Each field descriptor is variable-length:

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `offset` | Byte offset of this field within the aggregate |
| 4 | 4 | `type` | Type word |
| 8 | variable | `name` | ASD string |

The C or C++ tag name for a struct, class, or union is not represented in the debug table. Earlier ARM tools described union types as struct items whose field offsets were all zero.

For C and C++ bitfields, the type part of a field's type word identifies a bitfield item.

## 16. Array Item, Code 7

An array item describes a one-dimensional array. Multi-dimensional arrays are represented as arrays of arrays. Dimension order is source-language dependent and differs between C and Fortran.

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `header` | Code-and-length word, code 7 |
| 4 | 4 | `size` | Total byte size of the array |
| 8 | 4 | `flags` | Bound descriptor flags |
| 12 | 4 | `basetype` | Element type word |
| 16 | 4 | `lowerbound` | Constant bound value or variable-item offset |
| 20 | 4 | `upperbound` | Constant bound value or variable-item offset |

If `size` is zero, debugger operations that affect the whole array, rather than individual elements, are forbidden.

Bound flags:

| Mask | Name | Meaning |
| ---: | --- | --- |
| 1 | `ARRAY_UNDEF_LBOUND` | Lower bound is undefined |
| 2 | `ARRAY_CONST_LBOUND` | Lower bound is a constant |
| 4 | `ARRAY_UNDEF_UBOUND` | Upper bound is undefined |
| 8 | `ARRAY_CONST_UBOUND` | Upper bound is a constant |
| 16 | `ARRAY_VAR_LBOUND` | Lower bound is a variable |
| 32 | `ARRAY_VAR_UBOUND` | Upper bound is a variable |

For a constant bound, the corresponding bound field stores the bound value. For a variable bound, the corresponding bound field stores a section-relative offset to a variable item that describes the location containing the bound. That variable item may describe a compiler-known location that does not correspond to a source-language variable.

---

## 17. Subrange Item, Code 8

A subrange item describes a Pascal subrange type. It also serves to describe C enumerated types and Pascal scalars, where the base type is understood to be an unsigned integer of appropriate size.

PDF layout:

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `header` | Code-and-length word, code 8 |
| 4 | 4 | `lb` | Low bound of the subrange |
| 8 | 4 | `hb` | High bound of the subrange |
| 12 | 4 | `sizeandtype` | Container size and simple base type code |

Decode `sizeandtype` as:

```c
uint16_t container_size = sizeandtype & 0xffff; // 1, 2, or 4 bytes
uint16_t base_type_code = sizeandtype >> 16;    // simple type code
```

Example from the source: a subrange 256..511 of unsigned short may be held in one byte.

Source discrepancy: `ASDTF.txt` lists the fields as `sizeandtype`, `lb`, `hb`, while the ARM DUI 0041B PDF lists `lb`, `hb`, `sizeandtype`. This document follows the PDF. A tool intended to consume very early or independently generated ASD data should make this ordering configurable or validate against known producer behavior.

## 19. Enumeration Items, Codes 11 And 12

Enumeration items describe Pascal or C enumerated types.

### 19.1 Contiguous Enumeration, Code 11

Use this item when enumeration values form a contiguous range.

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `header` | Code-and-length word, code 11 |
| 4 | 4 | `type` | Type word describing the enumeration container |
| 8 | 4 | `count` | Cardinality of the enumeration |
| 12 | 4 | `base` | First and lowest enumeration value, possibly negative |
| 16 | variable | `nametable` | Sequence of `count` ASD strings |

Use this item when enumeration values are not contiguous.

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `header` | Code-and-length word, code 12 |
| 4 | 4 | `type` | Type word describing the enumeration container |
| 8 | 4 | `count` | Cardinality of the enumeration |
| 12 | variable | `nametable` | Sequence of `count` value/name pairs |

Each value/name pair is variable-length:

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |

## 20. Procedure/Function Declaration Item, Code 13

A declaration item describes a function or procedure signature.

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `header` | Code-and-length word, code 13 |
| 4 | 4 | `type` | Return type word |
| 8 | 4 | `argcount` | Number of arguments |
| 12 | variable | `args` | Sequence of `argcount` argument descriptors |

Each argument descriptor is variable-length:

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `type` | Argument type word |
| 4 | variable | `name` | ASD string |

## 22. Bitfield Item, Code 16

A bitfield item describes an individual bitfield member of a C or C++ struct, union, or class.

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `header` | Code-and-length word, code 16 |
| 4 | 4 | `type` | Type word describing the bitfield value type |
| 8 | 4 | `container` | Type word describing the containing storage type |
| 12 | 1 | `size` | Bitfield size in bits |
| 13 | 1 | `offset` | Offset of bit 0 of the field from bit 0 of the container |
| 14 | 2 | `zero` | Zero padding bytes |

The bit offset is measured from the least-significant bit of the container, independent of target byte order.

---

## 23. Macro Definition Item, Code 17

A macro definition item describes a C or C++ preprocessor `#define`.

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `header` | Code-and-length word, code 17 |
| 4 | 4 | `fileentry` | Section-relative offset of source file entry containing the macro definition |
| 8 | 4 | `sourcepos` | Source position of the macro definition |
| 12 | 4 | `body` | Section-relative offset of replacement text |
| 16 | 4 | `argcount` | Number of macro arguments, or -1 for object-like macros |
| 20 | 4 | `argtable` | Section-relative offset of argument-name table, or 0 if no arguments |
| 24 | variable | `name` | ASD string containing macro name |

The replacement body is not an ASD string. It is the replacement text as bytes terminated by a zero byte.

If `argcount > 0`, `argtable` points to `argcount` ASD strings containing argument names. If the macro has no arguments, `argtable` is zero. For object-like macros, `argcount` is the signed word value `-1`.

The `body` and `argtable` offset fields are physically stored in the macro definition item, but their values are offsets from the start of the containing ASD section.

| 4 | 4 | `fileentry` | Section-relative offset of source file entry containing the macro undefinition |

| 12 | variable | `name` | ASD string containing macro name |

## 25. Fileinfo Item, Code 10

A fileinfo item appears once per section after all other debug items. It describes source files and source-to-code line mappings.

The section item's `fileinfo` field points to the fileinfo item. If `fileinfo` is zero, no source file information is present.

If a fileinfo item is too large for the 16-bit item length field, its header length must be zero. Use the section `debugsize` to determine its extent.

Fileinfo layout:

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `header` | Code-and-length word, code 10 |
| 4 | variable | `file_entries` | Sequence of file entries |
| end | 4 | zero word | Terminates the file entry sequence |

Each file entry begins with a nonzero length word. A single zero word marks the end of the sequence.

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `len` | Length of this file entry in bytes |
| 4 | 4 | `date` | Last modification time, or 0 |
| 8 | variable | `filename` | ASD string, possibly empty |
| after filename | variable | `fragment_data` | Fragment count and fragment items |

`date`, when present, is seconds since the start of 1970, the Unix date origin. Zero means not available or unused.

`len` is measured from the start of the file entry's `len` word. Use it to locate the next file entry. Procedure, endproc, and macro `fileentry` fields point to the start of one of these file entries.

Fragment data layout:

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `n` | Number of following fragments |
| 4 | variable | `fragments` | Sequence of `n` fragment items |

Each source file is described as a sequence of fragments. A fragment represents a contiguous region of source within which compiled code addresses increase monotonically with source position. Fragment order is not necessarily source-position order. A compilation with no included files may have a single fragment.

### 25.1 Fragment Items

Each fragment begins with five words followed by a byte stream of lineinfo items.

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `size` | Length of this fragment in bytes, including lineinfo data |
| 4 | 4 | `firstline` | First source line number |
| 8 | 4 | `lastline` | Last source line number |
| 12 | 4 | `codestart` | Address of the fragment's first executable code byte |
| 16 | 4 | `codesize` | Byte size of code in the fragment |
| 20 | variable | `lineinfo` | Variable-length statement mapping data |

Use `size` to locate the next fragment. The source does not specify extra padding after the lineinfo stream; do not assume padding beyond what `size` includes.

### 25.2 Lineinfo Stream

Each lineinfo item describes one source statement. A decoder normally starts a fragment at source position `(firstline, 1)` and code address `codestart`, then advances both positions using each lineinfo item.

Short form, two bytes:

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 1 | `codeinc` | Bytes of code generated by this statement |
| 1 | 1 | `lineinc` | Source-space increment to next statement |

Short-form source advancement:

| Condition | Next source position |
| --- | --- |
| `0 <= lineinc < 64` | `(line + lineinc, 1)` |
| `lineinc >= 64` | `(line, column + lineinc - 64)` |

The code address always advances by `codeinc` bytes.

A statement may generate zero bytes of code if the source increment is nonzero. A statement with zero code increment and zero source increment is not representable because zero-increment encodings are used as long-form escapes.

Long forms are used if `codeinc` is greater than 255, if a line change greater than 63 is needed, or if a column change greater than 191 is needed.

Two-halfword long form, six bytes total, selected by initial bytes `codeinc = 0`, `lineinc = 0`:

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 1 | `codeinc_escape` | 0 |
| 1 | 1 | `lineinc_escape` | 0 |
| 2 | 2 | `lineinc` | Source-line increment |
| 4 | 2 | `codeinc` | Code-byte increment |

This form is used when the next statement starts in column 1. The next source position is `(line + lineinc, 1)`.

Three-halfword long form, eight bytes total, selected by initial bytes `codeinc = 0`, `lineinc = 64`:

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 1 | `codeinc_escape` | 0 |
| 1 | 1 | `lineinc_escape` | 64 |
| 2 | 2 | `lineinc` | Source-line increment |
| 4 | 2 | `codeinc` | Code-byte increment |
| 6 | 2 | `newcol` | Absolute column number for next statement |

This form is used when the next statement does not start in column 1. The next source position is `(line + lineinc, newcol)`.

Halfwords in long forms use target endianness. The long-form halfword order is `lineinc`, then `codeinc`, which is the reverse of the short-form byte order. In the three-halfword form, `newcol` is an absolute column number, not an increment.

Decode until the fragment byte position reaches `fragment_start + size`.

Version caution: the source notes that this lineinfo encoding is incompatible with a previous ASD format in which short-form `lineinc` always described a line increment and there was no column-aware long form. The same sources also state that `asdversion` should be 2. Tooling that must support older ASD producers should gate lineinfo interpretation on known producer/version behavior where available.

## 26. FP Map Fragment Item, Code 32

An FP map fragment item describes the offset from the stack pointer of a **virtual frame pointer** in functions that do not use a physical frame pointer. Stack offsets in variable items are then offsets from this virtual frame pointer.

Such functions can occur because a no-frame-pointer APCS variant was selected, or because simple leaf functions omit a physical frame pointer. FP map fragments are generated in a separate debug area regardless of whether other debug tables are generated.

Layout:

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 4 | `header` | Code-and-length word, code 32 |
| 4 | 4 | `bytes` | Exact byte size of this item |
| 8 | 4 | `codestart` | Address of first word of code described |
| 12 | 4 | `saveaddr` | Address of instruction saving callee-save registers, or 0 |
| 16 | 4 | `codesize` | Size of code area described |
| 20 | 4 | `initoffset` | Initial virtual-FP offset from SP |
| 24 | variable | `FPInfo` | Sequence of FPInfo records |

The item-header length field is rounded up to a word boundary, while `bytes` gives the exact item size. If `bytes` is not a multiple of four, bytes from `bytes` up to the item-header length are padding.

FPInfo short form, two bytes:

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 1 | `codeinc` | Unsigned size of code area described by this record |
| 1 | 1 | `offsetinc` | Signed change in virtual-FP offset for the next record |

FPInfo long form, six bytes, selected by two zero bytes:

| Offset | Size | Field | Description |
| ---: | ---: | --- | --- |
| 0 | 1 | `codeinc_escape` | 0 |
| 1 | 1 | `offsetinc_escape` | 0 |
| 2 | 2 | `codeinc` | Unsigned halfword code increment |
| 4 | 2 | `offsetinc` | Signed halfword offset increment |

A decoder starts at `codestart` with virtual-FP offset `initoffset`. Each FPInfo record describes `codeinc` bytes of code at the current offset, then adds `offsetinc` to obtain the offset for the next record.

## 27. Parser Algorithm

High-level parser outline:

```c
parse_asd_block(base, size, endian):
    pos = 0
    while pos < size:
        section = parse_section_item(base + pos)
        require section.code == 1
        require section.debugsize >= section.item_length
        require pos + section.debugsize <= size

        if section.lang == LANG_NONE:
            parse_low_level_payload(base + pos, section)
        else:
            parse_high_level_items(base + pos, section)

        pos += section.debugsize
```

High-level item loop:

```c
parse_high_level_items(section_base, section):
    pos = section.item_length
    end = section.debugsize

    while pos < end:
        h = read_u32(section_base + pos)
        length = h >> 16
        code = h & 0xffff

        if code == 10 && length == 0:
            length = end - pos

        require length >= 4
        require pos + length <= end

        if code is known:
            parse_item(section_base, pos, code, length)
        else:
            skip length bytes

        pos += length
```

Low-level section parser outline:

```c
parse_low_level_payload(section_base, section):
    nsyms = section.nsyms
    pos = section.item_length

    for i in 0..nsyms-1:
        sym = read_u32(section_base + pos)
        value = read_u32(section_base + pos + 4)
        pos += 8

    string_table_length = read_u32(section_base + pos)
    string_table = section_base + pos
    require pos + string_table_length <= section.debugsize
```

Do not parse low-level symbols as normal ASD items.

## 28. Writer Checklist

When writing ASD data:

- Use the target's byte order for all words and halfwords.
- Emit byte fields and strings in address order.
- Start every ASD section with a section item.
- Set each item header length to the byte size including the header.
- Pad ASD strings with zero bytes to a word boundary.
- Encode all internal references as byte offsets from the start of the containing ASD section.
- Emit relocatable address fields for all code and data addresses as required by the enclosing container.
- Set `section.debugsize` to the total byte size of the section.
- Emit fileinfo last when source-file information is present.
- If fileinfo is larger than 65535 bytes, write its item-header length as zero.
- Preserve source nesting by placing nested definitions between procedure and endproc items.
- For low-level sections, write raw symbol records and the AOF-style string table after the section item, not normal ASD item records.
- Set `asdversion` to 2 for tables following the documented format.

## 29. Validation Checklist

A robust reader should validate at least:

- Every section begins with item code 1.
- `section.debugsize` stays within the enclosing ASD block.
- Non-fileinfo item lengths are nonzero and at least 4.
- Known fixed-size items have the expected minimum length.
- Variable-length strings stay within their containing item.
- Section-relative offsets point within the current section.
- Type-word compound offsets point to plausible type-description items.
- Procedure `endproc` offsets either point to code 3 items or are zero for labels.
- Fileinfo, when present, is the final high-level item.
- File entry `len` values and fragment `size` values make forward progress and stay within fileinfo.
- Lineinfo decoding consumes exactly each fragment's declared `size`.
- Low-level string table length stays within the section and is a multiple of 4.
