# ARM Symbolic Debug Table Format

This document summarizes the ARM Symbolic Debug, or ASD, table format
documented in the ARM Software Development Toolkit 2.11a manuals in
this directory.

Primary source: `REFMAN.PDF`, Chapter 15, "ARM Symbolic Debug Table
Format".

Supporting sources: `REFMAN.PDF` sections on AIF, AOF, ELF, compiler,
and assembler options, plus `USRGUIDE.PDF` section 7.10.3 on accessing
debug information.

## Purpose

ASD tables contain symbolic debugging data generated by ARM compilers
and used by the ARM symbolic debugger for high-level,
language-oriented interactive debugging.

For each separate compilation unit, called a section in the ASD
specification, the compiler emits debugging data and a special
object-code area. The linker combines those debugging data areas into
one contiguous debug data block in the final image.

Low-level and source-level debugging are independent. An image can
contain either, both, or neither.

## Containers

### AOF Object Files

In ARM Object Format, an area with attribute bit 15 set contains
symbolic debugging tables. Debugging table areas are normally
read-only, so bit 13 should also be set. The linker groups these areas
so they can be accessed as one continuous chunk before or at runtime.

### AIF Images

In ARM Image Format, an uncompressed image layout can contain optional
debugging data after the read-only and read-write areas.

The AIF header has an `Image Debug size` field and an `Image debug
type` field:

| Value | Meaning |
| --- | --- |
| 0 | No debugging data present |
| 1 | Low-level debugging data present |
| 2 | Source-level ASD debugging data present |
| 3 | Low-level and source-level ASD debugging data present |

Code and data references from debugging tables are relocatable
addresses. After the image is loaded at its load address, these values
are effectively absolute. References between debug table entries are
byte offsets from the start of the debugging data area, so the
debugging data is position-independent after image relocation.

A debugger is expected to copy debugging data to safe storage before
the debuggee overwrites it with zero-initialized data, heap, or stack
data.

### ELF Executables

ARM executable ELF supports ASD debugging tables for compatibility
with ARM's Symbolic Debugger. ASD debugging information is stored in a
single ELF section named `.asd`.

## Generation Options

The C compiler uses ASD table format by default with `-asd`. It can
use DWARF instead with `-dwarf`.

Compiler debug information controls include:

| Option | Meaning |
| --- | --- |
| `-asd` | Use ASD table format, the default |
| `-dwarf` | Use DWARF table format |
| `-g+` | Enable generation of debug tables for the current compilation |
| `-g-` | Disable generation of debug tables for the current compilation |
| `-gt` | Generate all available source-level debug table entries |
| `-gtp` | Do not include preprocessor macro definitions in ASD tables |
| `-gx` | No optimizations while generating debug tables |
| `-gxr` | Unoptimized register allocation |
| `-gxo` | Full optimization, with limitations described by the manual |

The assembler `-g` option outputs ARM Symbolic Debugger debugging
tables suitable for `armsd` and the ARM Debugger for Windows.

The linker includes available debug information by default. `-Nodebug`
suppresses debug tables.

For ROM debugging where a plain binary image cannot carry debug
information, the user guide describes producing a matching `-bin -aif`
image containing debug data, then loading only symbols with `armsd
-symbols filename` or `readsyms filename`.

## Terminology And Encoding

| Term | Size | Notes |
| --- | --- | --- |
| byte | 8 bits | Usually unsigned |
| halfword | 16 bits | Used mainly by long line information and FP map data |
| word | 32 bits | Often considered signed |

The object file and debug data use the byte order of the intended
target, not necessarily the host. Producers and consumers must treat
words and halfwords according to target endianness. Byte sequences
such as strings are written and read in address order without endian
conversion.

## Debug Data Structure

An ASD debug data area is a sequence of items. The first item in each
debug area is a section item. Other items follow in source-definition
order, preserving nesting. Procedure bodies are bracketed by procedure
and endproc items. The final item in a section is the fileinfo item,
when source-file information is present.

The section item contains a `debugsize` field giving the total byte
length of the debug data for that section. Offsets stored in items are
byte offsets from the start of that section's debug data, meaning from
the start of the section item.

## Item Header

Every debug item starts with a 32-bit code-and-length word:

| Bits | Meaning |
| --- | --- |
| 31..16 | Byte length of the item |
| 15..0 | Item code |

If a debugger encounters an unknown item code, it should use the
length field to skip the item. This allows later extensions without
invalidating existing debuggers.

For a fileinfo item whose length cannot fit in 16 bits, the item
length field is written as zero. This is valid because fileinfo is the
final item and the section item carries the total section length.

## Item Codes

| Code | Item |
| --- | --- |
| 1 | Section |
| 2 | Procedure or function definition |
| 3 | Endproc |
| 4 | Variable |
| 5 | Type |
| 6 | Struct |
| 7 | Array |
| 8 | Subrange |
| 9 | Set |
| 10 | Fileinfo |
| 11 | Contiguous enumeration |
| 12 | Discontiguous enumeration |
| 13 | Procedure or function declaration |
| 14 | Begin naming scope |
| 15 | End naming scope |
| 16 | Bitfield |
| 17 | Macro definition |
| 18 | Macro undefinition |
| 19 | Class |
| 20 | Union |
| 32 | FP map fragment |

## Strings

String fields are encoded as a length byte followed by string
bytes. The string is padded with zero bytes to a word boundary. The
string length range is 0 to 255 bytes.

## Source Positions

Several items contain a `sourcepos` word:

| Bits | Meaning |
| --- | --- |
| 31..22 | Character offset from the start of the line, zero-based |
| 21..0 | Line number |

## Type Words

Several items contain a type word:

| Bits | Meaning |
| --- | --- |
| 31..8 | Base type code |
| 7..0 | Pointer count |

Pointer count values encode the type directly:

| Value | Meaning |
| --- | --- |
| 0 | The type itself |
| 1 | Pointer to the type |
| 2 | Pointer to pointer to the type |

Simple base type codes are decimal:

| Code | Type |
| --- | --- |
| 0 | void |
| 10 | signed byte |
| 11 | signed halfword |
| 12 | signed word |
| 13 | signed double word |
| 20 | unsigned byte |
| 21 | unsigned halfword |
| 22 | unsigned word |
| 23 | unsigned double word |
| 30 | float |
| 31 | double |
| 32 | long double |
| 41 | single complex |
| 42 | double complex |
| 100 | function |

Compound types such as arrays and structures are represented by
separate debug items. The type code is negative, and its negation is
the byte offset of the compound-type item from the start of the debug
area.

## Section Item

A section item is the first item of each debug data section. After the
code-and-length word, it contains four flag bytes:

| Byte | Meaning |
| --- | --- |
| `lang` | Source language code |
| `flags` | Level of detail |
| `unused` | Unused byte |
| `asdversion` | Debug data version |

Language byte codes:

| Code | Name | Meaning |
| --- | --- | --- |
| 0 | `LANG_NONE` | Low-level debugging data only |
| 1 | `LANG_C` | C source-level debugging data |
| 2 | `LANG_PASCAL` | Pascal source-level debugging data |
| 3 | `LANG_FORTRAN` | Fortran-77 source-level debugging data |
| 4 | `LANG_ASM` | ARM assembler line-number data |

All other language codes are reserved by ARM.

The flags byte describes whether debugging data contains line-number
information, top-level variable information, or both. The PDF text
identifies these as mask values but does not expose numeric mask
constants.

The `asdversion` byte should be set to 2 for this definition.

The flag bytes are followed by word-sized fields:

| Field | Meaning |
| --- | --- |
| `codestart` | Address of first instruction in this section, relocated by the linker |
| `datastart` | Address of static data for this section, relocated by the linker |
| `codesize` | Byte size of executable code in this section |
| `datasize` | Byte size of static data in this section |
| `fileinfo` | Offset of the section's fileinfo item, or 0 if absent |
| `debugsize` | Total byte length of debug data for this section |
| `name` or `nsyms` | Source-level name string, or low-level symbol count |

For C, Pascal, and Fortran, `name` is a string. For low-level
debugging data, `lang == 0`, this field is a 4-byte integer `nsyms`.

### Linker-Generated Low-Level Data

For linker-generated low-level debug data, the section item values
are:

| Field | Value |
| --- | --- |
| `language` | 0 |
| `codestart` | `Image$$RO$$Base` |
| `datastart` | `Image$$RW$$Base` |
| `codesize` | `Image$$RO$$Limit - Image$$RO$$Base` |
| `datasize` | `Image$$RW$$Limit - Image$$RW$$Base` |
| `fileinfo` | 0 |
| `nsyms` | Number of following symbols |
| `debugsize` | Total low-level debug data size including this section item |

The section item is followed by `nsyms` symbol items. Each symbol item
has two words:

| Field | Meaning |
| --- | --- |
| `sym` | Flags plus byte offset in string table of symbol name |
| `value` | Symbol value |

The 24 least significant bits of `sym` encode the string-table
index. The most significant bits encode flags:

| Flag | Value | Meaning |
| --- | --- | --- |
| `ASD_ABSSYM` | 0 | Symbol is absolute |
| `ASD_GLOBSYM` | `0x01000000` | Symbol is global |
| `ASD_TEXTSYM` | `0x02000000` | Symbol names code |
| `ASD_DATASYM` | `0x04000000` | Symbol names data |
| `ASD_ZINITSYM` | `0x06000000` | Symbol names zero-initialized data |
| `ASD_16BITSYM` | `0x10000000` | Symbol is a Thumb symbol |

The linker reduces all symbol values to absolute values. The flag
values record the symbol's origin in the image.

Immediately after the symbol table is a string table in standard AOF
format. It contains a length word followed by NUL-terminated
strings. The length word includes itself, so no string-table offset is
less than 4. The string table is padded with NULs to a word boundary.

## Procedure Or Function Definition Item

A procedure item appears once for each procedure or function
definition. Nested definitions appear between the procedure item and
its matching endproc item.

Fields after the code-and-length word:

| Field | Meaning |
| --- | --- |
| `type` | Return type for a function, otherwise 0 |
| `args` | Number of arguments |
| `sourcepos` | Source position of procedure start |
| `startaddr` | First instruction of the procedure prologue |
| `entry` | First instruction after the prologue and first sensible high-level breakpoint location |
| `endproc` | Offset of the related endproc item |
| `fileentry` | Offset of the source file-list entry containing the procedure start |
| `name` | String |

A source label is represented as a special procedure item with no
matching endproc. In that case `endproc` is 0. Pascal and Fortran
numeric labels are converted to strings prefixed by `$n`. For
Fortran-77, multiple entry points to the same procedure produce
separate procedure items that share the same endproc offset.

## Type Item

A type item describes a named source-language type, such as a C
`typedef`.

| Field | Meaning |
| --- | --- |
| `type` | Type word |
| `name` | String |

## Endproc Item

An endproc item marks the end of the items belonging to a procedure
and gives return information.

| Field | Meaning |
| --- | --- |
| `sourcepos` | Source position of procedure end |
| `endpoint` | Address of the code byte after the procedure's compiled code |
| `fileentry` | Offset of the file-list entry for the procedure end |
| `nreturns` | Number of procedure return points, possibly 0 |
| `retaddrs` | Array of addresses of procedure return code |

If the procedure body is an infinite loop, `nreturns` is 0. Otherwise,
each return address should identify a point where an exit breakpoint
can be set while the current stack frame still belongs to the
procedure.

## Variable Item

A variable item describes a source variable or formal argument. The
first variable items inside a procedure describe its arguments.

| Field | Meaning |
| --- | --- |
| `type` | Type word |
| `sourcepos` | Source position of the variable |
| `storageclass` | Variable storage class |
| `location` | Location, interpreted by storage class |
| `name` | String |

Storage classes:

| Code | Meaning |
| --- | --- |
| 1 | External variable, or Fortran common |
| 2 | Static variable private to one section |
| 3 | Automatic variable |
| 4 | Register variable |
| 5 | Pascal `var` argument |
| 6 | Fortran argument |
| 7 | Fortran character argument |

The `location` field depends on storage class:

| Storage class | Location meaning |
| --- | --- |
| Static or external | Absolute address, relocated by linker |
| Automatic or Pascal `var` argument | Stack offset from frame pointer |
| Fortran argument | Offset into argument list |
| Register variable | Register number, with floating-point registers numbered 16 to 23 |

The debugger uses `sourcepos` to distinguish definitions with the same
name in disjoint source-level naming scopes.

## Struct, Union, And Class Items

Struct, union, and class items have the same format. The C or C++ tag
name for a struct, union, or class is not represented in the debug
table.

Fields after the code-and-length word:

| Field | Meaning |
| --- | --- |
| `fields` | Number of fields |
| `size` | Total byte size |
| `fieldtable` | Array of field items |

Each field item contains:

| Field | Meaning |
| --- | --- |
| `offset` | Byte offset of this field within the aggregate |
| `type` | Type word |
| `name` | String |

Older ARM tools described unions as struct items where all fields have
offset 0. For C and C++ bitfields, the type part of the field's type
word identifies a bitfield item.

## Array Item

An array item describes a one-dimensional array. Multi-dimensional
arrays are arrays of arrays. Dimension order is source-language
dependent.

| Field | Meaning |
| --- | --- |
| `size` | Total byte size of array |
| `flags` | Bound descriptor flags |
| `basetype` | Type word for element type |
| `lowerbound` | Constant value or variable location |
| `upperbound` | Constant value or variable location |

If `size` is zero, debugger operations affecting the whole array,
rather than individual elements, are forbidden.

Array flag values:

| Flag | Value | Meaning |
| --- | --- | --- |
| `ARRAY_UNDEF_LBOUND` | 1 | Lower bound is undefined |
| `ARRAY_CONST_LBOUND` | 2 | Lower bound is a constant |
| `ARRAY_UNDEF_UBOUND` | 4 | Upper bound is undefined |
| `ARRAY_CONST_UBOUND` | 8 | Upper bound is a constant |
| `ARRAY_VAR_LBOUND` | 16 | Lower bound is a variable |
| `ARRAY_VAR_UBOUND` | 32 | Upper bound is a variable |

Bounds can be undefined, constant, or variable. For a constant bound,
the bound field is the value. For a variable bound, the bound field is
the offset of a variable item describing the location that contains
the bound.

## Subrange Item

A subrange item describes a Pascal subrange type. It is also used for
C enumerated types and Pascal scalars.

| Field | Meaning |
| --- | --- |
| `lb` | Low bound |
| `hb` | High bound |
| `sizeandtype` | Container size and base type |

In `sizeandtype`, the least significant 16 bits encode byte size of
the container, 1, 2, or 4. The most significant 16 bits encode a
simple type code.

## Set Item

A set item describes a Pascal set type partially.

| Field | Meaning |
| --- | --- |
| `size` | Byte size of the object |

## Enumeration Items

A contiguous enumeration item contains:

| Field | Meaning |
| --- | --- |
| `type` | Type word describing the container |
| `count` | Cardinality of the enumeration |
| `base` | First, lowest, value |
| `nametable` | Character array containing `count` name strings |

A discontiguous enumeration item contains:

| Field | Meaning |
| --- | --- |
| `type` | Type word describing the container |
| `count` | Cardinality of the enumeration |
| `nametable` | Table of `count` value-name pairs |

Each discontiguous `nametable` entry contains the enumerated value
followed by a string name.

## Function Declaration Item

A function declaration item describes the signature of a function or
procedure.

| Field | Meaning |
| --- | --- |
| `type` | Return type word |
| `argcount` | Number of arguments |
| `args` | Sequence of argument description items |

Each argument description item contains a type word and a string
name. An argument descriptor can be unnamed. In that case, the name
length is zero and the name field is a single zero word.

## Begin And End Naming Scope Items

These items mark the beginning and end of a naming scope. They must be
properly nested.

| Field | Meaning |
| --- | --- |
| `codeaddress` | Address of the start or end of the scope |

## Bitfield Item

A bitfield item describes a C or C++ bitfield member of a struct,
union, or class.

| Field | Meaning |
| --- | --- |
| `type` | Type word describing the field |
| `container` | Type word describing the field's container |
| `size` | Byte containing the bitfield size in bits |
| `offset` | Byte containing offset from bit 0 of the containing value to bit 0 of the field |
| padding | Two zero bytes to align to a word boundary |

## Macro Definition Item

A macro definition item describes a C or C++ preprocessor `#define`.

| Field | Meaning |
| --- | --- |
| `fileentry` | Offset of the source file-list entry containing the macro definition |
| `sourcepos` | Source position of the macro definition |
| `body` | Offset of the replacement text, terminated by a zero byte |
| `argcount` | Number of macro arguments, or -1 for object-type macros |
| `argtable` | Offset of argument-name strings, or 0 if no arguments |
| `name` | String |

The `body` and `argtable` offsets are stored inside the macro
definition item, but the offsets are relative to the containing
section.

## Macro Undefinition Item

A macro undefinition item describes a C or C++ preprocessor `#undef`.

| Field | Meaning |
| --- | --- |
| `fileentry` | Offset of the source file-list entry containing the macro undefinition |
| `sourcepos` | Source position of the macro undefinition |
| `name` | String |

## Fileinfo Item

A fileinfo item appears once per section, after all other debug
items. Each source file is described as a sequence of fragments. A
fragment describes a contiguous region of source where code addresses
increase monotonically with source position. Fragment order is not
necessarily source-position order.

After the code-and-length word, the fileinfo item contains a sequence
of file entry items followed by a single zero word.

Each file entry item contains:

| Field | Meaning |
| --- | --- |
| `len` | Length of this file entry in bytes, including fragment data |
| `date` | Last modification time as seconds since the start of 1970, or 0 if unavailable or unused |
| `filename` | String, or empty string if unknown |
| `fragment data` | Fragment count and fragment items |

Fragment data contains:

| Field | Meaning |
| --- | --- |
| `n` | Number of fragments |
| `fragments` | Sequence of `n` fragment items |

Each fragment item contains five words followed by lineinfo entries:

| Field | Meaning |
| --- | --- |
| `size` | Length of this fragment in bytes, including lineinfo entries |
| `firstline` | First line number |
| `lastline` | Last line number |
| `codestart` | Pointer to the fragment's executable code start |
| `codesize` | Byte size of the fragment code |
| `lineinfo` | Variable-length source-to-code mapping data |

### Lineinfo Encoding

Each lineinfo item describes one source statement. It starts as two
unsigned bytes:

| Byte | Meaning |
| --- | --- |
| `codeinc` | Bytes of code generated by this statement |
| `lineinc` | Source-space occupied by this statement |

Short form interpretation:

| Condition | Next source position |
| --- | --- |
| `0 <= lineinc < 64` | `(line + lineinc, 1)` |
| `lineinc >= 64` | `(line, column + lineinc - 64)` |

If `codeinc` is greater than 255, or if the line or column change
cannot fit in the short form, both bytes are written to describe zero
increments and the real values follow as halfwords.

The two-halfword long form is used when the next statement starts in
column 1:

| Field | Meaning |
| --- | --- |
| `zero` | Two zero bytes |
| `lineinc` | Halfword source-line increment |
| `codeinc` | Halfword code-byte increment |

The three-halfword long form is used when the next statement does not
start in column 1:

| Field | Meaning |
| --- | --- |
| escape bytes | `codeinc = 0`, `lineinc = 64` |
| `lineinc` | Halfword source-line increment |
| `codeinc` | Halfword code-byte increment |
| `newcol` | Halfword absolute column number for the next statement |

The long-form halfword order is `lineinc` then `codeinc`, the reverse
of the short-form byte order.

A statement can generate zero bytes of code if its line increment is
non-zero. A zero-code, zero-line increment item is not representable
because the zero-zero encoding is reserved as a long-form escape.

The lineinfo encoding is incompatible with a previous ASD
format. Consumers should interpret lineinfo according to the table
format version recorded in the section item.

## FP Map Fragment Item

An FP map fragment item describes offsets from the stack pointer of
the virtual frame pointer in functions that do not use a physical
frame pointer. Stack offsets in variable items are then relative to
the virtual frame pointer.

FP map fragments are generated in a separate debug area regardless of
whether other debug tables are generated.

Fields after the code-and-length word:

| Field | Meaning |
| --- | --- |
| `bytes` | Exact size of the item in bytes; the item-header length is rounded up to a word boundary |
| `codestart` | Address of first word of code described |
| `saveaddr` | Address of instruction saving callee-save registers, or 0 if none in the area |
| `codesize` | Size of code area described |
| `initoffset` | Offset of virtual FP from SP at the start of the code area |
| `FPInfo` | Variable sequence of FPInfo records |

An FPInfo record is either two bytes:

| Field | Meaning |
| --- | --- |
| `codeinc` | Unsigned byte size of code area described by this item |
| `offsetinc` | Signed byte change in virtual FP offset for the next item |

Or, if the byte form cannot represent the values, two zero bytes
followed by two halfwords:

| Field | Meaning |
| --- | --- |
| escape | Two zero bytes |
| `codeinc` | Unsigned halfword |
| `offsetinc` | Signed halfword |

## Parsing Checklist

1. Locate the ASD data in the container: AOF debugging-table area, AIF
   debug data with type 2 or 3, or ELF `.asd` section.
2. Determine target endianness from the enclosing object or image
   format.
3. Start at a section item and use `debugsize` to bound that section's
   debug data.
4. For each item, decode the code-and-length word. Skip unknown item
   codes by length.
5. Decode string fields as length-prefixed byte strings padded to word
   alignment.
6. Resolve offsets relative to the section item address, not relative
   to the current item.
7. Interpret relocated code and data addresses as absolute after final
   image relocation.
8. Treat the fileinfo item as the final item. If its item-header
   length is zero, use the enclosing section's `debugsize` to find its
   extent.
9. Resolve type words by simple code for primitive types and by
   negative offset for compound type items.
10. Interpret lineinfo according to the section item's ASD version.

## Known Gap In The Source Manual

The section item's `flags` byte is described as having mask values for
line-number information, top-level variable information, and both. In
the provided `REFMAN.PDF`, the numeric mask constants for these flags
are not visible in extracted text or word-position extraction. An
implementation that must write ASD tables should confirm those
constants against original headers or another authoritative ASD
specification.
